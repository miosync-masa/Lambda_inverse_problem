# Lambda³ NNNU — 産業向け技術概要

## ■ 何を作ったか

ニューラルネットワークを一切使わない異常検知フレームワーク「Lambda³ NNNU」。物理学由来の構造テンソル理論に基づき、センサー時系列から「正常とは何か」を数学的に構造化し、その構造からの逸脱を検知する。

検知原理は一つだけ：**正常を構造化し、逸脱を通知する。異常パターンは一切学習しない。**

国際ベンチマーク（NAB: Numenta Anomaly Benchmark、52 ファイル 6 カテゴリ）において、公開ベンチマーク主要手法（HTM 70.5、CAD OSE 69.9、Bayes Changepoint 68.9 — いずれもオンライン学習型）を上回るスコア **72.02** を達成。特に産業システム由来の時系列カテゴリ（realKnownCause: 機械温度、CPU 利用率、データセンター障害、需要時系列等の混成 7 ファイル）では **78.30**、AWS 系インフラ監視データ（realAWSCloudwatch、16 ファイル）では **77.54** を記録。

---

## ■ 何ができるか

### 突発異常のリアルタイム検知（Tier 0 ストリーミング）

- センサー信号のフレーム単位リアルタイム監視
- 6 種の独立した統計的スコアラー（jump / gradual / drift / reconstruction / kernel / structural）による OR 投票方式
- パラメータチューニング不要。NAB の 52 ファイル（異種多様）に同一設定で適用し 72.02 を達成
- ニューラルネット不要のため、推論段階はエッジデバイスでも動作可能

### 運転モード対応の高精度検知（Tier 2 レジーム対応）

- センサー値の統計的クラスタを自動識別（GMM + BIC 自動選択、K ∈ [1, 5]）
- 各クラスタが「アイドル」「低負荷」「高負荷」「過渡期」等の物理的意味を持つかは、現場エンジニアが事後解釈する設計（システム側はクラスタ番号で扱う）
- クラスタごとに正常の閾値を個別設定 → 「高負荷運転時にしては異常」と「アイドル時にしては異常」を区別可能
- 正常データのみで学習。異常データは「除外」にのみ使用し、異常の形状は一切学習しない
- 工場の実務フロー（事後的に異常期間をタグ付け → ベースラインから除外）にそのまま適合

### 多チャンネル独立検知と故障モード推定（拡張予定）

- 振動・温度・歪み・電流等、各センサーに 1 つの NNNU インスタンスを配置し完全独立で検知
- どのセンサーの組み合わせが flag を立てたかから、ルールベースで故障モードを推定
- 故障ルールは YAML/JSON で現場エンジニアが設定変更可能。ブラックボックス化しない
- 本機能は現状未実装、PoC フェーズで実装予定

### 長期ドリフト検知（拡張予定）

- シフト単位・日単位の定期バッチで中央値の推移を監視
- 経年劣化、季節変動など、リアルタイム層では構造的に拾えない長期変化を補完
- アラートではなく定期レポートとして報告。突発検知との責務分離
- 本機能は現状未実装、PoC フェーズで実装予定

## ■ 何ができないか（honest framing）

### 長期季節変動の即時検知

通年のエアコンデータのように、正常の全レジームを観測するのに数ヶ月かかるケースでは、ストリーミング層だけでは対応できない。上記の長期ドリフト検知（バッチ層）で補完する設計。

キャリブレーション期間に全運転モードが含まれるほど検知精度が向上する。**正常データは多いほど良い**。

### 異常の種類の自動分類

NNNU は「正常からの逸脱」を検知するだけであり、「何が壊れたか」は検知しない。

故障モード推定は上位の診断レイヤー（MultiChannelDiagnostics、拡張予定）がルールベースで行う。ルールは現場のドメイン知識に依存し、自動生成されない。

### 未知の故障モードの予測

ルールテーブルにない組み合わせは `unknown_fault` として報告する。これは限界であると同時に、新規故障モード発見の手がかりでもある。

### 多チャンネル間の相互作用の検知

各センサーは独立に検知する。「振動と電流の位相関係が変化した」のような相互作用は、現状のアーキテクチャでは検知対象外。

## ■ 何がわかるのか

### フレーム単位

- 各センサーの異常スコア（連続値）と二値判定（正常/異常）
- どのスコアラー（jump / gradual / drift / reconstruction / kernel / structural）が反応したかの内訳 → 異常の性質の手がかり（突発的か、漸進的か、構造変化か）

### 運転モード単位

- 現在どの統計的クラスタ（運転レジーム）にいるかの自動分類
- レジームごとの正常閾値 → 文脈付きの判定が可能

### 多チャンネル統合（拡張後）

- どのセンサーの組み合わせが同時に flag を立てたか → 故障モードの推定
- 重症度指標（flag 立ちチャンネル数 / 全チャンネル数）

### 長期トレンド（拡張後）

- シフト間・日間の中央値変化率 → 劣化の進行速度の定量把握
- 「先週と今週でベースラインが 5% 上昇している」といった傾向レポート

## ■ どんなメリットがあるのか

### 導入・運用コスト

- **ニューラルネットワーク不要** → GPU サーバー不要、推論段階はエッジデバイスで動作可能
- **パラメータチューニング不要** → NAB の 52 ファイル（CPU 利用率・温度・ネットワーク・取引データなど異種多様）を同一設定で評価し 72.02 を達成。同型機への横展開時も、基本的に正常データさえ与えれば再チューニング不要と期待される（PoC で確認予定）
- **正常データのみで学習** → 異常データの収集・ラベリングコストがゼロ。正常運転期間のデータを与えるだけ

### 現場との親和性

- **（拡張後）ルールベースの故障モード推定** → 現場エンジニアが理解・修正・拡張できる設計。「AI が何を考えているかわからない」問題が発生しない
- **運転モード自動識別** → 「この運転状態にしては異常」という文脈付きの判定。誤報削減に直結
- **honest failure（正直な失敗）** → 検知できないときは中途半端な結果を出さず明示的に失敗する（NAB 実測で、検知可能なファイルは 95+ / 検知不可能なファイルは 0-30 の二極化分布）。産業現場で最も危険な「自信ありげな誤判定」を構造的に排除

### スケーラビリティ

- 各センサーが完全独立 → 並列計算が自明。チャンネル数に比例するだけ
- **（拡張後）同型機の正常構造（GMM パラメータと per-regime threshold）を機群間で共有することで、1 台だけ逸脱した場合の早期発見が可能になる設計**。本機能は現状未実装、PoC で実装予定
- 時間解像度の拡張（フレーム〜週次）が同一原理（NNNU 原理：正常を構造化 → 逸脱を検知）で統一 → 手法の乱立による保守負荷を回避

### 知的財産・差別化

- **Lambda³ 理論**（物理学由来の構造テンソル理論）に基づく独自手法。具体的には、時系列の遅延埋め込み（delay embedding）と低ランク部分空間（SVD top-k）への射影残差を「構造からの逸脱」として定式化。scikit-learn / PyOD 等の汎用 OSS とは数学的構造が異なる
- **国際ベンチマーク（NAB）で HTM 超えを実証済み** → 技術的信頼性の客観的根拠
- **「ニューラルネットを使わない」こと自体が、説明可能性・監査可能性が要求される産業現場での競争優位**

## ■ 計算性能（参考値）

- **推論**: per-frame `O(n_cal × d)` 程度の軽量演算（NN 不要）
  - 例: `n_cal = 2000`、`d = 5` → 約 10 KFLOPS / frame
  - 最も重い Kernel scorer でも上記水準、エッジデバイスで秒間数千フレーム処理可能
- **キャリブレーション**: 初期 1 回のみ実施
  - GMM fit + 6 scorer baseline 構築
  - CPU 1 コアで数秒〜数十秒（データ量による）
- **メモリ**: per-channel 約 50 KB（GMM パラメータ + per-regime threshold）
- **スケーラビリティ**: チャンネル数に対し線形（完全独立並列化）

## ■ アーキテクチャ全体像

```
                各センサー (N 個、独立)
                       │
   ┌───────────────────┼────────────────────┐
   │                   │                    │
   ▼                   ▼                    ▼
NNNU Streaming     NNNU Regime-aware    BatchDriftDetector
(Tier 0)            (Tier 2)            (拡張予定)
                                          │
   per-channel独立     per-channel独立     per-channel独立
   フレーム単位        運転モード単位      シフト/日単位
   │                   │                    │
   │ binary+score      │ binary+score       │ DriftReport
   │                   │                    │
   └─────────┬─────────┴────────────────────┘
             │
             ▼
   MultiChannelDiagnostics (拡張予定)
             │
             ▼
   DiagnosisReport
     - flagged_channels: ['vibration', 'temperature']
     - fault_mode:       'lubrication_failure'
     - confidence:       0.50  (2/4 ch flag)
     - channel_scores:   {...}
```

全レイヤーが同一の NNNU 原理（正常を構造化 → 逸脱を検知）で動作。BatchDrift は時間スケール、MultiChannel はチャンネル次元、それぞれの拡張軸が違うだけ。**どのレイヤーも異常パターンを学習しない**。

## ■ PoC 提案例

実機データでの段階的検証を推奨：

| Phase | 内容 | 必要データ | 検証項目 |
|---|---|---|---|
| **Phase 1** | 単一センサー検証 | 振動 1 ch、健全〜異常までの実機データ（数日〜数週間） | Tier 0 / Tier 2 の検知精度比較、誤報率測定 |
| **Phase 2** | 多チャンネル統合 | 4 ch（振動・温度・歪み・電流）の同期記録 | MultiChannelDiagnostics 実装と故障モード推定の精度検証 |
| **Phase 3** | 長期運用 | 1〜3 ヶ月の連続稼働データ | BatchDriftDetector による経年劣化トレンド検出 |
| **Phase 4** | 機群展開 | 同型機 5〜10 台の同期記録 | 機群間正常構造共有による相対異常検知（1 台だけ逸脱の早期発見） |
